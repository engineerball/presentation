What is a Two-Tower Model?

Two neural networks (towers) that independently encode queries and candidates into a shared embedding space.

score(user, item) = u · v = Σᵢ uᵢ · vᵢ → high = good match

Understanding Embedding Space

Both towers map inputs to the same D-dimensional vector space. Semantically similar entities cluster together.

User Features

  • Interaction history (views, likes, purchases)
  • Search queries
  • Location, language
  • Demographics, preferences

Item Features

  • Title, description, category
  • Publisher, language
  • Dynamic stats (views, likes)
  • Content embeddings (text, image)

Key insight: The training pairs define the geometry of the embedding space. Whatever semantic relationship positive pairs encode becomes the similarity metric.

Contrastive Learning Objective

Train to pull positive pairs together and push negative pairs apart in embedding space.

Positive Pairs

(user, item) actually interacted

→ Minimize distance → high dot product

Examples: clicked, watched, purchased

Negative Pairs

(user, item) never interacted

→ Maximize distance → low dot product

Examples: random items, negative impressions

Loss = Σ positive ‖u − v‖² − Σ negative ‖u − v_neg‖²

Goal: maximize margin between positive and negative similarities

Loss Functions for Two-Tower Training

Loss Mechanism Use Case
MSE (Regression) Predict interaction magnitude; minimize squared error Simple, interpretable; Red Hat OpenShift AI approach
Log Loss (BCE) Binary classification: positive vs negative Pointwise; explicit pos/neg pairs
BPR Pairwise: rank positive above negative Ranking-focused; no magnitude needed
InfoNCE (Contrastive) Multi-class: positive among negatives; temperature τ In-batch negatives; most popular in practice

InfoNCE: L = −log[ exp(u·v⁺/τ) / Σᵢ exp(u·vᵢ/τ) ]

Lower τ = sharper signal (focus on hardest negatives); higher τ = softer, more stable

Decoupled Inference — The Key Advantage

Unlike joint models, two towers compute embeddings independently — enabling massive scale optimization.

Offline (Precompute)

  • All candidate item embeddings computed once
  • ANN index built from precomputed embeddings
  • Incremental updates when new items arrive
  • New items get embeddings too — cold start solved
⟹

Online (Serving)

  • Only run user tower at query time
  • Single embedding → ANN lookup
  • Millions of items searched in < 100ms
  • Serving is O(log N), not O(N)

Result: Serving cost is constant regardless of corpus size — only query tower runs, then index lookup.

User Encoder Architecture

Processes diverse user signals into a single vector. Each feature type has its own processing pipeline.

Interaction History

Watched items + timestamps → sequence encoding → pooled embedding

Red Hat: BGE model (384→58D per attribute → averaged)

Search Queries

Past searches → Text vectorization → embedding

Captures intent beyond interaction history

Contextual Features

Location, language, device, time of day → embedding

Normalizes numerical features before projection

Output

Concatenated → Dense(512) → Dense(256) → u ∈ ℝᵈ

Red Hat: 64D; Google: 1280D; production varies

Google's YouTube DNN: watch-time features outperform click-through for preventing clickbait

Item Encoder Architecture

Mirrors the user tower structure but processes item-specific features. Must handle both static and dynamic attributes.

Google Cloud feature processing:

  • Text: TextVectorization (n-grams) → Embedding → GlobalAveragePooling1D
  • Categorical: Hashing (faster) or StringLookup (accurate) → Embedding
  • Numeric: Normalize → pass through

Feature Engineering & Magnitude Scoring

Not all interactions are equal. Transform raw signals into meaningful training targets.

InteractionMagnitude Score Design
🛒 PurchaseScaled by quantity
⭐ RatingInverse mapping (bad=high, good=low)
👁️ View/ClickBase magnitude (lowest signal)
🛕 Cart AddMedium magnitude (intent signal)

Training Data Design

  • Positive pairs = <query, candidate> that interacted
  • Use features as they were before the interaction (not current features)
  • This prevents data leakage in the training signal

Important: The choice of feature pair determines the embedding space — <playlist, track> yields title-space; <audio features, audio features> yields audio-space.

Feature Processing Trade-offs

Key engineering decisions that impact model quality and training speed.

Hashing

  • Faster: O(1) computation
  • Risk: Hash collisions at small bin sizes
  • No vocabulary scan needed
  • Good for high-cardinality features

StringLookup

  • No collisions — exact mapping
  • Cost: Requires vocabulary adaptation scan
  • Slower for very large vocabularies
  • Better accuracy, needs preprocessing

Dense + Cross Layers: model explicit feature interactions before dense layers

Improves accuracy at computational cost — evaluate parallel vs. stacked configurations

Why Negative Sampling Is Critical

Recommendation data has only implicit positives — everything else is unobserved. Without negatives, the model learns trivial solutions.

Without negatives: The model learns to output "+1" for every user-item pair — useless for ranking.

Strategy 1: Random Negatives

  • Randomly pick items not watched by the user
  • ✅ Simple, unbiased
  • ✅ Works best in initial training stages
  • ⚠️ "Easy" negatives — obvious mismatches

Strategy 2: Negative Impressions

  • Items shown but not clicked/watched
  • ✅ Harder, more realistic negatives
  • ⚠️ Risk: Confuses the model on borderline cases
  • ⚠️ "Second-best" Sci-fi gets same weight as irrelevant romcom

Best practice: Start with random negatives. Once high recall is achieved, add some negative impression training. Let the reranking layer learn from negative impressions instead of embeddings.

In-Batch Negatives

The most efficient negative sampling approach in modern training — free negatives from batch structure.

How it works:

A batch of 256 (user, item) pairs creates a 256×256 similarity matrix. The diagonal is positive pairs; all off-diagonal entries are negatives.

Pros:

  • Extremely efficient — no extra sampling
  • Natural fit with InfoNCE / contrastive losses
  • Scales with batch size automatically

Cons:

  • Negative quality depends on batch composition
  • False negatives if same item appears multiple times
  • Popular items → more likely false negatives

InfoNCE: L = −log[ exp(u·v⁺/τ) / Σᵢ exp(u·vᵢ/τ) ]

τ (temperature) controls sharpness — tuned as hyperparameter

Hard Negative Mining & False Negatives

The hardest negatives provide the strongest gradient signal — but they can be wrong.

Mitigations

  • Mix hard + random negatives
  • Filter items with high predicted relevance
  • Deduplicate within batches

False Negative Problem

  • User would like the item but hasn't seen it
  • Common with popular items & hard negatives
  • Contaminates training signal

Loss Function Negative Strategy Paradigm Characteristics
Log Loss (Binary Cross-Entropy) Random / Popularity-based Pointwise Simple, explicit pos/neg pairs
BPR (Bayesian Personalized Ranking) Random / Popularity-based Pairwise Rank pos above neg; no magnitude needed

Industry trend: InfoNCE with in-batch negatives dominates modern production systems due to efficiency and strong performance.

Approximate Nearest Neighbor (ANN) Serving

Precomputed item embeddings are indexed for fast retrieval. Exact search is too slow — ANN provides the speed/recall tradeoff.

HNSW

Hierarchical Navigable Small World

  • Graph-based multi-layer structure
  • Good recall, moderate memory
  • Used by FAISS, Pinecone, Milvus

Tree-AH (Google)

Tree + Asymmetric Hashing

  • Tree clusters embedding space hierarchically
  • AH approximates dot product scoring
  • Scales to 1B+ vectors

Recall vs Latency tradeoff: Increasing leaf nodes searched % ↑ recall, ↑ latency. This is the most critical tuning knob.

Brute-force index provides baseline for measuring real ANN recall (not theoretical)

Incremental Index Updates & Cold Start

The system doesn't need full retraining for new items or users — the decoupled architecture handles this naturally.

New Items (Cold Start)

  • Feed item features through candidate tower → get embedding
  • Insert into ANN index (incremental update, no rebuild)
  • Item gets recommended even with zero interaction history
  • Described in terms of known items via shared embedding space

New Users (Cold Start)

  • Collect category preferences during onboarding
  • Show popular items in selected categories (short-term)
  • Encoder creates embedding from registration data (medium-term)
  • Selected items saved as positives for future training

Google Cloud: Vertex AI Vector Search supports incremental updates — no full rebuild needed. Autoscaling handles load spikes.

Practical Tips & Best Practices

Data Quality

  • Use feature snapshots (as they were before interaction)
  • Watch-time > click-through for preventing clickbait
  • Magnify signal with quantity data (e.g., # purchases)
  • Clean negative impressions from noise

Training

  • Start with random negatives, add impression-based later
  • Profile training with Cloud Profiler / TensorBoard
  • Fully utilize GPU/TPU — they're the costliest component
  • Vectorize data transformations over batches

Serving

  • Tune ANN: leaf nodes % is the biggest recall lever
  • Deploy both ANN + brute-force index for recall measurement
  • Precompute all candidate embeddings offline
  • Use boolean filtering + crowding tags for diversity

Implementation

  • Use TFRS or TensorFlow before building from scratch
  • Evaluate recall vs latency tradeoff honestly
  • Consider embedding dimensions (64D–1280D range)
  • Plan for monitoring embedding drift over time

Key Takeaways

1. Two towers map queries & items to shared embedding space — dot product = relevance score

2. Decoupled inference enables scale — precompute candidate embeddings, only run query tower at serving

3. Negative sampling strategy shapes learning — random first, impressions later, let reranking handle hard cases

4. Side information is the key advantage — unlike matrix factorization, two towers incorporate text, images, metadata

5. ANN index is the serving bottleneck — tune leaf nodes % to balance recall vs latency; use brute-force as baseline

Two-tower retrieval is the foundation every personalization system should build first.
