What is a Two-Tower Model?

Two neural networks that independently encode queries and candidates into a shared embedding space.

score(user, item) = u · v = Σᵢ uᵢ · vᵢ → high = good match

Understanding Embedding Space

Both towers map inputs to the same D-dimensional vector space. Semantically similar entities cluster together.

User Features

  • Interaction history (views, likes, purchases)
  • Search queries
  • Location, language
  • Demographics, preferences

Item Features

  • Title, description, category
  • Publisher, language
  • Dynamic stats (views, likes)
  • Content embeddings (text, image)

Key insight: The training pairs define the geometry of the embedding space.

Contrastive Learning Objective

Train to pull positive pairs together and push negative pairs apart in embedding space.

Positive Pairs

(user, item) actually interacted
→ Minimize distance → high dot product
Clicked, watched, purchased

Negative Pairs

(user, item) never interacted
→ Maximize distance → low dot product
Random items, negative impressions

Loss = Σ positive ‖u − v‖² − Σ negative ‖u − v_neg‖²

Goal: maximize margin between positive and negative similarities

Loss Functions for Two-Tower Training

LossMechanismUse Case
MSE (Regression)Predict interaction magnitude; minimize squared errorSimple, interpretable
Log Loss (BCE)Binary classification: positive vs negativePointwise; explicit pos/neg pairs
BPRPairwise: rank positive above negativeRanking-focused; no magnitude needed
InfoNCE (Contrastive)Multi-class: positive among negatives; temperature τIn-batch negatives; most popular

InfoNCE: L = −log[ exp(u·v⁺/τ) / Σᵢ exp(u·vᵢ/τ) ]

Lower τ = sharper signal; Higher τ = softer, more stable

Decoupled Inference — The Key Advantage

Unlike joint models, two towers compute embeddings independently — enabling massive scale optimization.

Offline (Precompute)

  • All candidate item embeddings computed once
  • ANN index built from precomputed embeddings
  • New items get embeddings → cold start solved
⟹

Online (Serving)

  • Only run user tower at query time
  • Single embedding → ANN lookup
  • Millions of items searched in < 100ms

Result: Serving cost is constant regardless of corpus size.

Training Data Design

  • Positive pairs = <query, candidate> that interacted
  • Use features as they were before the interaction (prevents data leakage)

Feature pair choice determines embedding space — <playlist, track> vs <audio feature, audio feature>

Feature Processing Trade-offs

Key engineering decisions that impact model quality and training speed.

Hashing

  • Faster: O(1) computation
  • Risk: Hash collisions at small bin sizes
  • No vocabulary scan needed
  • Good for high-cardinality features

StringLookup

  • No collisions — exact mapping
  • Cost: Requires vocabulary adaptation scan
  • Slower for very large vocabularies
  • Better accuracy, needs preprocessing

Dense + Cross Layers: model explicit feature interactions before dense layers

Improves accuracy at computational cost — evaluate parallel vs. stacked

Why Negative Sampling Is Critical

Recommendation data has only implicit positives. Without negatives, the model learns trivial solutions.

Strategy 1: Random Negatives

  • Randomly pick items not watched by the user
  • ✅ Simple, unbiased
  • ✅ Works best in initial stages
  • ⚠️ "Easy" negatives — obvious mismatches

Strategy 2: Negative Impressions

  • Items shown but not clicked/watched
  • ✅ Harder, more realistic
  • ⚠️ Risk: Confuses model on borderline cases
  • ⚠️ "2nd-best" Sci-fi same weight as irrelevant genre

Best practice: Start with random. Once high recall achieved, add some impression-based. Let reranking learn from impressions.

In-Batch Negatives

Most efficient negative sampling — free negatives from batch structure.

A batch of 256 (user, item) pairs → 256×256 similarity matrix. Diagonal = positive pairs. All off-diagonal = negatives.

Pros:

  • Very efficient — no extra sampling
  • Natural fit with InfoNCE
  • Scales with batch size

Cons:

  • Negative quality depends on batch
  • False negatives if item repeats
  • Popular items → more false negatives

InfoNCE: L = −log[ exp(u·v⁺/τ) / Σᵢ exp(u·vᵢ/τ) ]

τ (temperature) controls sharpness

Hard Negative Mining & False Negatives

The hardest negatives give the strongest gradient — but they might be wrong.

Mitigations:

  • Mix hard + random negatives
  • Filter items with high predicted relevance
  • Deduplicate within batches

False Negative Problem:

  • User would like item but hasn't seen it
  • Common with popular items & hard negatives
  • Contaminates training signal

Approximate Nearest Neighbor (ANN) Serving

Precomputed item embeddings are indexed for fast retrieval. Exact search is too slow — ANN provides speed/recall tradeoff.

HNSW

  • Graph-based multi-layer structure
  • Good recall, moderate memory
  • Used by FAISS, Pinecone, Milvus

Tree-AH (Google)

  • Tree clusters embedding space
  • AH approximates dot product scoring
  • Scales to 1B+ vectors

Recall vs Latency: Increasing leaf nodes searched % ↑ recall, ↑ latency. The most critical tuning knob.

Incremental Updates & Cold Start

New Items

  • Feed item features through candidate tower → embedding
  • Insert into ANN index (incremental, no rebuild)
  • Recommended even with zero interaction history
  • Described via learned embedding space

New Users

  • Collect category preferences during onboarding
  • Show popular items in selected categories
  • Encoder creates embedding from registration data
  • Selected items → positives for future training

Google Cloud: Vertex AI Vector Search supports incremental updates — no full rebuild needed.

Practical Tips & Best Practices

Data Quality

  • Use feature snapshots (before interaction)
  • Watch-time > click-through for clickbait prevention
  • Magnify signal with quantity data

Training

  • Start with random negatives, add impressions later
  • Profile training with Cloud Profiler / TensorBoard
  • Fully utilize GPU/TPU — costliest component

Serving

  • Tune ANN: leaf nodes % is biggest recall lever
  • Deploy ANN + brute-force for recall measurement
  • Precompute all candidate embeddings offline

Implementation

  • Use TFRS or TensorFlow before building from scratch
  • Evaluate recall vs latency honestly
  • Plan for monitoring embedding drift over time

Key Takeaways

1. Two towers map queries & items to shared embedding space — dot product = relevance

2. Decoupled inference enables scale — precompute candidate embeddings, only query tower at serving

3. Negative sampling strategy shapes learning — random first, impressions later, reranking for hard cases

4. Side information is the key advantage — unlike matrix factorization, incorporates text, images, metadata

5. ANN index is the bottleneck — tune leaf nodes % for recall vs latency

Two-tower retrieval is the foundation every personalization system should build first.
