  FinOps คืออะไร?

FinOps = Financial Opserations — การทำงานร่วมกันระหว่างทีม Finance, Engineering, Product เพื่อให้เกิด ความรับผิดชอบทางการเงิน ในโมเดล cloud ที่เปลี่ยนตามการใช้งาน

"FinOps ไม่ใช่การลดต้นทุน — แต่คือการทำให้ ทุกเหรียญที่เราใช้ ได้ value สูงสุด"

  ทำไมต้อง FinOps?

  ปัญหาหลัก: Cloud bill พุ่งสูง แต่ไม่มีใครรู้ ว่าทำไม และ ทีมไหนใช้เท่าไร

  ไม่มี FinOps

  • Bill มาทีหลัง รู้ตัวอีกทีก็แพงแล้ว
  • ไม่มีใครรับผิดชอบค่า cloud
  • ทีม Eng ไม่เห็น impact ทางเงิน
  • Finance ทำ budget ไม่ตรงกับจริง

  มี FinOps

  • เห็นต้นทุน แบบ real-time
  • แต่ละทีม รู้ต้นทุนของตัวเอง
  • ตัดสินใจ architectural decision พร้อม data
  • CFO + CTO คุยกัน ในข้อมูลชุดเดียวกัน

  Key insight: FinOps เปลี่ยนจาก "ใครใช้เท่าไร?" เป็น "เราคุ้มหรือยัง?"

  FinOps ≠ ลดงบ แต่คือ Value

❌ ไม่ใช่: ลด cost
→
✅ ใช่: เพิ่ม value / unit cost

  ลด budget อย่างเดียว = กับดัก

ถ้าลดโดยไม่ได้ดู value → performance ตก, user experience เสีย, innovation หยุดชะงัก

  FinOps = เพิ่มหน่วยธุรกิจ value

ดู cost per unit of business เช่น cost/customer, cost/transaction, cost/feature — ไม่ใช่แค่ total bill

  Unit Economics > Aggregate Spend — ดูประสิทธิภาพ ไม่ดูยอดรวม

  6 หลักการ FinOps

North Star Principles · ทุกการตัดสินใจยึดหลักนี้

  หลักการที่ 1-2: Collaboration + Value

1.   ทีมต้องร่วมมือกัน

Finance, Eng, Product, Leadership — ทำงานร่วมกันในระดับที่เหมาะสม

ตัวอย่าง: สัปดาห์ละ meeting 30 min ระหว่าง FinOps + Tech Lead + PM เพื่อ review cost trend

🤝 ไม่ร่วมมือ = Eng รู้แต่ spec ไม่รู้ราคา | Finance รู้แต่ราคา ไม่รู้ spec

2.   Business value ขับเคลื่อน

Unit economics > aggregate spend — ตัดสินใจจาก value ที่ได้

ตัวอย่าง: ตัดสินใจย้ายจาก on-demand → CUDs เพราะเห็น trade-off cost vs flexibility

🎯 มี value = เพิ่ม spend ได้ ถ้าได้ business value มากกว่าที่จ่าย

  หลักการที่ 3-4: Ownership + Data

3.   ทุกคนเป็นเจ้าของ

ทีม Eng รับผิดชอบ cost ของตัวเอง ตั้งแต่ design → deploy → operate

ตัวอย่าง: ทุก Pull Request มี cost impact review ถ้าเปลี่ยน architecture

🙋 Anti-pattern: "cost ไม่ใช่เรื่องฉัน — finance manage ให้"

4.   ข้อมูลชัด เปิดทุกคน

Cost visibility แบบ real-time ทุกคนเข้าถึงข้อมูลชุดเดียวกัน

ตัวอย่าง: Dashboard กลาง ที่ทุกทีมเข้าดูได้ — ไม่เก็บ cost ไว้คนเดียว

📈 Fast feedback loop = พฤติกรรมเปลี่ยนเร็วขึ้น เห็น cost ทันที ไม่ใช่สิ้นเดือน

  หลักการที่ 5-6: Central Enablement + Variable Cost

5.   Centralized Enablement

ทีม FinOps กลาง — ทำ rate opt, best practice — ไม่ใช่ทำ optimization ให้ทุกทีม

ตัวอย่าง: ทีมกลาง negotiate enterprise discount 30% ให้ทั้ง org — แต่ทีม Eng เลือกขนาด VM เอง

🛡️ Economies of scale — ซื้อ bulk discount ได้เพราะมองทั้ง org

6.   ใช้ข้อดี Variable Cost

Cloud = จ่ายตามใช้ — รับข้อดีนี้ ใช้ agile planning แทน fixed budget

ตัวอย่าง: แทนที่จะจอง capacity ล่วงหน้า 1 ปี → ใช้ auto-scaling + spot instances

⚡ Variable ≠ ไม่แน่นอน — แต่ปรับได้ตาม demand จริง ลด over-provisioning

  Maturity Model: Crawl → Walk → Run

Crawl

เริ่มต้น — เห็นภาพ — ตั้ง labels, dashboard เบื้องต้น, หา quick wins

ตัวอย่าง: สร้าง GCP project ทุกโปรเจกต์, ตั้ง label team, env ดู bill 2 ครั้ง/เดือน, หา idle VM 5 ตัวแรกที่ปิดได้

Walk

ทำเป็นระบบ — ปรับปรุง — มี process, automate monitoring, มี KPI cost per unit

ตัวอย่าง: ทุกทีมมี cost dashboard ตัวเอง, monthly review meeting, set budgets + alerts 80%, CUDs ครอบคลุม 70% spend

Run

เป็นนิสัย — วัด value — cost เป็น first-class metric, automate optimization, unit economics-driven

ตัวอย่าง: CI/CD pipeline มี cost check gate, auto-scaling tuned real-time, cost per transaction KPI, FinOps score 90%+

  อย่าข้ามขั้น: Crawl ก่อนแล้วค่อย Walk — การข้ามขั้นอาจทำให้ data ไม่ได้มาตรฐาน วัดผลไม่ได้

  Phase 1: Inform — "รู้"

ก่อนจะ optimize ได้ — เราต้องรู้ก่อนว่าเงินไปไหน

  Cost Allocation

Dashboard real-time — ทุกทีมเห็น cost ของตัวเอง

ตัวอย่าง: GCP Billing Export → BigQuery → LookerStudio

จัดสรร shared cost แบบ fair — network, logging, monitoring

ตัวอย่าง: แบ่งตาม % usage, pro-rated by team

  Forecasting

ทำนาย budget ล่วงหน้า — ตั้ง budget alerts

ตัวอย่าง: alert เมื่อ cost เกิน 80% ของ budget เดือนนี้

  ตัวอย่าง: Labeling Strategy ที่ดี

Label Key ตัวอย่าง Values ทำไมต้องมี
team backend, frontend, data-eng รู้ว่าทีมไหนใช้ cloud เท่าไร
env dev, staging, prod แยก cost prod (ห้ามปิด)与非prod (ปิดได้)

cost-center cc-1001, cc-2002 ตรงกับ finance accounting

  Best practice: ใช้ Organization Policy บังคับ — สร้าง resource ถ้าไม่มี label หลัก → reject

  ตัวอย่าง: CUDs vs On-demand

Committed Use Discounts (CUDs) — สัญญา 1-3 ปี แลกกับ discount สูง

On-Demand CUD 1 ปี CUD 3 ปี
n2-standard-4 $0.1919/hr $0.1343/hr (-30%) $0.0960/hr (-50%)
Monthly (730 hr) $140 $98 $70
50 Instances $7,000 $4,900 $3,500
ประหยัด/เดือน — -$2,100 -$3,500

  ข้อควรระวัง: CUDs = commitment! ถ้า traffic ลดลง ต้องจ่ายอยู่ — ใช้เฉพาะ workload ที่ stable

  Best practice: Start with 1-year CUDs ครอบคลุม 60-70% baseline → ปลอดภัยกว่า 3-year lock-in โดยยังไม่รู้ trend

  ตัวอย่าง: Budget Alerts + Guardrails

ป้องกัน Bill Shock — แจ้งเตือนก่อนเงินหมด!

  Alerts — แจ้งเตือน

  • 50%: หมายเหตุ — "กลางเดือนแล้ว ครึ่ง budget หมดแล้ว"
  • 80%: ⚠️ Warning — ทีมต้อง review ทันที
  • 100%: 🚨 Critical — แจ้ง management พร้อม plan
  • 125%: 🔴 Emergency — auto-quota หรือ freeze ใหม่

  Guardrails — ป้องกัน

  • Quotas: จำกัด max VM instances per team
  • Org Policies: ห้ามสร้าง GPU instances โดยไม่ approve
  • Auto-shutdown: dev/staging env ปิด 22:00-07:00
  • Approval flows: spend > $1K/เดือน ต้องมี manager approve

  ตัวอย่าง Incident: Team Delta ทำ batch job ลูปไม่หยุด → VM 50 instances เปิด 72 ชม. → $15K ใน 3 วัน! Budget alert ที่ 80% แจ้งแล้ว แต่ทีมไม่เห็น → ต้อง set PagerDuty alert ด้วย

  Blaming Culture

  • "ใครทำ?!" → คนทำกลัว ไม่รายงาน
  • ปัญหาซ่อน — ครั้งหน้าเกิดอีก
  • วัฒนธรรมไม่ปลอดภัย → innovation ตาย

  Blameless Culture

  • "ระบบไหนพลาด?" → หา root cause
  • แก้ที่ process — ตั้ง hard quota
  • คนรายงานปัญหา → เรียนรู้ทั้งทีม

  Action items จาก Incident: (1) ตั้ง hard quota $5K/project/เดือน (2) PagerDuty alert เมื่อ cost spike >200% ใน 24 ชม. (3) batch job template auto-timeout 4 ชม.

  Key Takeaways

1. FinOps ≠ ลดงบ → เพิ่ม value ต่อ unit cost

2. ทุกคนเป็นเจ้าของ — ไม่ใช่ทีม FinOps เดียว

3. Inform → Optimize → Operate — ทำซ้ำไม่มีวันเสร็จ

4. Labels/Tags = จุดเริ่มต้น — ไม่มี = มองไม่เห็น

5. Unit economics > Total bill

6. Centralized enablement, decentralized execution

7. Blameless culture — เรียนรู้ ไม่โทษ

8. Crawl → Walk → Run — อย่าข้ามขั้น
