What is a Semantic Layer?

A virtual translation layer between the physical data warehouse and data consumers — BI tools, AI agents, notebooks, custom apps. Instead of writing SQL against raw tables, consumers query business concepts.

Physical

Tables, columns, joins, data types

→

Semantic

Entities, metrics, dimensions, filters

"A semantic layer lies between business users and the database, letting them compose queries in the concepts they understand." — Julian Hyde

Physical vs Semantic — The Mindset Shift

Physical LayerSemantic (Logical) Layer
Datasets / tablesAggregations over fact tables
Columns with technical namesMetrics with business definitions
Foreign key joinsRelationships with declared cardinality
SQL query stringsLogical queries → auto-rewritten to SQL
Schema changes break everythingVersioned model definitions (Git)

Key insight: Ad-hoc queries need semantic flexibility. Change granularity daily→weekly→monthly, add a region, roll up to country — all instantly. A semantic layer handles this on-the-fly without ETL jobs.

The Problem: Metric Drift

  Metric drift: the same KPI produces different values depending on which tool you ask

Root cause: business logic embedded inside individual BI tools instead of centralized

Anti-Pattern

  • "Revenue" defined in DAX and LookML and dbt
  • Active user = 7-day in Sales, 30-day in Marketing
  • Churn calculation hardcoded in dashboards
  • Every tool reinvents governance

Solution

  • Define metrics once in a version-controlled model
  • Version-controlled, peer-reviewed definitions
  • Single source queried by all downstream tools
  • Governance travels with the asset

"The correct answer depends on where you ask the question" — this is unacceptable at scale

Six Core Components

Dimensions

Who, what, where, when — categorical & temporal axes for slicing data

Measures

Computed business functions — sums, counts, ratios, rolling windows; grain-independent

Relationships

Declared joins enriching fact tables; supports star and snowflake schemas

Filters

Business rules baked into metric definitions — "active clients only", "last 90 days"

Governance

Ownership, descriptions, certification, lineage, row/column-level security

Performance

Pre-aggregation, intelligent caching, smart routing to pre-computed results

Model Definitions (Git)

Query path: Consumer request → Semantic API → Query compiler (joins, filters, aggregations) → Optimized SQL pushdown to warehouse → Result → Cache for next time

Query Resolution Flow

1
Consumer sends logical query (e.g., "MRR by region, last quarter") via REST / GraphQL / SQL
2
Model parser resolves metric names to physical columns, dims to tables, applies declared joins and filters from model definition
3
Query compiler generates optimized SQL — selects appropriate pre-aggregations if available, determines grain, applies date filters
4
Optimizer checks aggregate cache — if a matching materialized result exists, returns it; otherwise pushes down to warehouse
5
Result delivery — returns structured result with metadata (metric definitions, certification status, lineage)

  Aggregate awareness: the engine intercepts every query, matches it against pre-built materialized aggregates, and rewrites to use the smallest/fastest pre-computed table

Three Modern Semantic Layer Types

Metrics-as-Code

Portable declarative format (YAML/SQL) compiled to SQL by a dedicated engine. Git-versioned, CI/CD-friendly.

Tools: dbt SL (MetricFlow), MetriQL

Headless / Universal

Standalone API service sitting between the warehouse and all consumers. Decoupled from any specific BI tool.

Tools: Cube, AtScale

Platform-Native

Semantics co-located with governance inside the data platform. Inherits access controls, lineages automatically.

Tools: Databricks UC, Snowflake

Trend: Snowflake (Semantic Views + Autopilot 2026) and Databricks (UC Metric Views) are pushing platform-native as the dominant pattern for enterprises that already use one platform.

Case Study: E-Commerce with dbt + BigQuery

Scenario: An e-commerce company uses BigQuery as warehouse, dbt for transformations. They need consistent metrics across Looker (marketing), Excel (finance), and a customer-facing dashboard.

Step 1: dbt Models in BigQuery

fct_orders

ColumnType
order_idSTRING
customer_idSTRING
order_dateDATE
order_totalNUMERIC
payment_statusSTRING
countrySTRING

fct_subscriptions

ColumnType
subscription_idSTRING
customer_idSTRING
mrr_monthDATE
mrr_amountNUMERIC
subscription_statusSTRING
plan_typeSTRING

One row per order / one row per customer-month. Clean analytics-ready tables from raw staging.

Step 2: Define Semantic Models

Map BigQuery tables to business entities with measures and dimensions:

# models/marts/ecommerce_metrics.yml
version: 2

semantic_models:
  - name: orders
    description: "E-commerce order transactions."
    model: ref('fct_orders')
    defaults:
      agg_time_dimension: order_date

    measures:
      - name: order_total
        agg: sum
        expr: order_total
      - name: completed_orders
        agg: count
        expr: order_id
        filters:
          - field: payment_status
            operator: '='
            value: "'completed'"

    dimensions:
      - name: order_date
        type: time
        type_params: { time_granularity: day }
      - name: customer_id, country
        type: categorical

The semantic model maps physical BigQuery columns → business measures + dimensions. Filters (e.g., "completed only") are baked in.

Step 3: Define the Metrics

Reusable, version-controlled metric definitions — the single source of truth:

metrics:
  - name: total_revenue
    label: "Total Revenue"
    description: "Sum of all completed order amounts."
    type: simple
    type_params:
      measure: order_total

  - name: average_order_value
    label: "AOV"
    description: "Average value of a completed order."
    type: ratio
    type_params:
      numerator: order_total
      denominator: completed_orders

  - name: monthly_recurring_revenue
    label: "MRR"
    description: "Total active subscription MRR."
    type: simple
    type_params:
      measure: mrr_amount
    filters:
      - field: subscription_status
        operator: '='
        value: "'active'"

Three metric types: simple (direct aggregation), ratio (numerator / denominator), derived (expression combining other metrics)

Step 4: How It Works — From Request to BigQuery

Business user asks: "Total Revenue by country, last quarter" via Looker:

1
Looker sends request to dbt SL API (GraphQL / JDBC):
get_metrics(metrics=['total_revenue'], group_by=['country'], where="order_date >= '2026-01-01'")
2
MetricFlow compiles → resolves total_revenue → sum(order_total) from fct_orders, adds WHERE payment_status = 'completed'
3
Generates optimized BigQuery SQL:
SELECT country, SUM(order_total) AS total_revenue
FROM `project.dataset.fct_orders`
WHERE payment_status = 'completed'
  AND order_date >= '2026-01-01'
GROUP BY country
4
BigQuery executes → leverages partition pruning on order_date, returns results to Looker

Pre-aggregation in Production

With 50M+ orders, scanning fct_orders for every query is too expensive. Pre-aggregate common patterns:

Pre-agg #1: Revenue by Month × Country

CREATE TABLE mart_agg.revenue_month_country
AS
SELECT
  DATE_TRUNC(order_date, MONTH) AS order_month,
  country,
  SUM(order_total) AS revenue,
  COUNT(order_id) AS order_count
FROM `dataset.fct_orders`
WHERE payment_status = 'completed'
GROUP BY 1, 2

Impact

50M rows Without pre-agg
→ ~15s, ~$0.50/query
3.6K rows With pre-agg (3y × 200 countries)
→ <100ms, ~$0.01/query

30-50x cost reduction for common queries

dbt SL / MetricFlow automatically selects the smallest matching pre-aggregation via "aggregate awareness" — no manual query routing needed.

Engineering Mechanics

Pre-aggregation, caching, query optimization, monitoring

Pre-Aggregation Deep Dive

Pre-aggregation is the #1 performance optimization. Materialize expensive queries upfront, answer from cached aggregates.

Rollup Aggregations

Pre-computed GROUP BY on common dimension combinations.
rollup(measure, dim1, dim2, time_dim → month)

OLAP Cube Pre-computation

Multi-dimensional cubes — every cell is pre-calculated. AtScale generates optimal cube sets.

Materialized Views

Warehouse-native materialized views. Auto-refresh scheduled. Pushdown-compatible with query rewrite.

Smart Pre-Aggregation Selection

Engine analyzes: query patterns, data volume, refresh frequency, storage budget. Generates optimal set automatically.

Aggregate Awareness

The semantic layer intercepts every query and determines: "Can I answer this from a pre-built aggregate instead of scanning raw data?"

1
Query arrives: "Revenue by month, region = 'EU'" — needs aggregation over millions of rows
2
Aggregate matching: Engine scans catalog of pre-aggregations — finds a rollup covering revenue × month × region
3
Rewrite: Query is rewritten against the smallest matching aggregate — from scanning 50M rows to 5K rows
4
Execute: Runs against the materialized aggregate table → sub-100ms response

Coverage problem: The "rollup lattice" grows exponentially with dimensions. Smart engines use heuristics: top-N query patterns, cardinality ratios, refresh cost — to generate a minimal sufficient set of pre-aggregations.

Query Rewriting & Pushdown Optimization

The semantic layer rewrites the logical query into optimized SQL, then pushes as much work as possible down to the warehouse.

Filter Pushdown

WHERE clauses pushed into innermost subqueries. Filters from metric definitions automatically applied.

Aggregation Pushdown

GROUP BY + aggregates executed on warehouse side. Semantic layer only resolves cross-database joins.

Join Optimization

Join graph traversal — picks shortest path, avoids unnecessary joins, uses materialized join tables.

Dialect Translation

Single model definition → different SQL dialects (BigQuery, Snowflake, Redshift, Spark SQL) automatically.

Key insight: the semantic layer is a query compiler — it takes high-level metric requests and compiles them to low-level SQL, using model definitions as the "type system" and "standard library."

Semantic Data Modeling Best Practices

How data engineers should structure the physical layer below the semantic layer for optimal consumption.

Star Schema

Entity-oriented fact tables with dimension tables. The semantic layer maps to this naturally — one fact table = one cube.

Business Names

AI-friendly dimension and measure names — descriptive, consistent. Avoid technical jargon in exposed metric names.

Documentation as AI Metadata

Every metric needs: description, unit, owner, grain, update frequency, certification level. LLMs use this for grounding.

Surrogate Time Dimensions

Every grain gets a time dimension. Enables time-series analysis, period-over-period, and rolling windows natively.

DBT + Semantic layer workflow: staging → intermediate marts (dbt-models.yml) → semantic layer (dbt_metrics.yml) → consumers. Each layer is version-controlled and tested.

AI & Agentic Analytics

Why the semantic layer is critical infrastructure for LLM-driven analytics

"Dangerously Plausible" — The LLM Problem

  Text-to-SQL against raw tables generates coherent-sounding but potentially incorrect, inconsistent, ungoverned answers

?
"What was our churn rate last quarter?"

The LLM must guess:

  • Which table? Is customer status in users or subscriptions?
  • What is "churn" — cancellation in 30 days? 60 days? Non-renewal?
  • "Last quarter" — calendar Q? Rolling 90d? Fiscal Q?
  • Active vs. cancelled — should trial users count?

Without a semantic layer: The LLM infers all of this from table names and column names. It might be right. It might be confidently wrong. You can't tell.

Semantic Layer as AI Grounding Infrastructure

The semantic layer provides the structured business context that LLMs need to generate accurate queries:

1
Grounding Phase: Agent reads semantic context — available metrics, definitions, synonyms, governance rules, certification status
2
Execution Phase: Agent queries through governed metric APIs via REST/GraphQL/SQL — never raw tables

Implementation Roadmap

Recommended rollout — avoid big-bang adoption:

1
Start small — one high-stakes metric + key dimensions. Prove value before scaling.
2
Integrate into existing tools — connect semantic layer to 1-2 existing dashboards. Show "same data, same number" to build trust.
3
Establish governance process — define metric ownership, certification workflow, peer review for definition changes.
4
Scale to additional domains — expand metrics catalog, add teams, optimize pre-aggregation strategy.
5
AI integration — connect AI agents to semantic layer for natural language analytics. Add synonym mapping, grounding metadata.

  Physical layer prerequisite: Ensure data models are clean — staging → intermediate → marts. The semantic layer can only be as good as the physical layer underneath it.
