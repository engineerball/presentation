Financial Operations สำหรับ Cloud

  FinOps คืออะไร?

FinOps คือแนวปฏิบัติและรูปแบบการดำเนินงาน ที่นำ ความรับผิดชอบทางการเงิน เข้าสู่รูปแบบค่าใช้จ่ายแบบแปรผันของ Cloud

— FinOps Foundation

วัฒนธรรม

ทุกทีมมีส่วนร่วมในการบริหารต้นทุน ไม่ใช่แค่ฝ่าย IT

ข้อมูล

ตัดสินใจจากข้อมูลที่แม่นยำ ทันเวลา และโปร่งใส

คุณค่า

เป้าหมายคือ คุณค่าสูงสุด ต่อทุกบาทที่ใช้ ไม่ใช่แค่ลดค่าใช้จ่าย

  ผู้เกี่ยวข้อง: IT, การเงิน, จัดซื้อ, ผู้บริหาร ทำงานร่วมกัน

  กระบวนการ: ทำซ้ำต่อเนื่อง ไม่ใช่โครงการครั้งเดียว

  FinOps ≠ การลดค่าใช้จ่าย

  ความเข้าใจที่ผิด

  • FinOps คือการ "หั่น" งบประมาณ Cloud
  • เป้าหมายคือใช้น้อยที่สุดเท่าที่เป็นไปได้
  • IT เป็นผู้รับผิดชอบเพียงฝ่ายเดียว
  • ทำครั้งเดียวแล้วเสร็จ

  ความเข้าใจที่ถูกต้อง

  • FinOps คือการ ใช้ให้ได้คุณค่าสูงสุด
  • เป้าหมายคือสมดุลระหว่างต้นทุนและประโยชน์
  • ทุกทีมมีส่วนร่วมในการบริหารต้นทุน
  • ปรับปรุงต่อเนื่องเป็นวงจร

  หลักการ: บางครั้งการใช้จ่ายเพิ่มขึ้นอาจเป็นสิ่งที่ถูกต้อง หากนำไปสู่คุณค่าทางธุรกิจที่มากกว่า

DGS 9-2:2568 & Go Cloud First

  นโยบาย Go Cloud First & DGS 9-2:2568

  คณะรัฐมนตรีมีมติเมื่อ 11 กันยายน 2566 ประกาศ "นโยบายการใช้คลาวด์เป็นหลัก (Go Cloud First)" สำหรับหน่วยงานภาครัฐทุกแห่ง

  DGS 9-2:2568 กำหนด

  • FinOps เป็น รูปแบบการดำเนินงานที่บังคับ สำหรับการใช้ Cloud ภาครัฐ
  • Pay-per-use เป็นรูปแบบการจัดซื้อ ลำดับแรก
  • ต้องวิเคราะห์ TCO ก่อนตัดสินใจเลือก Private Cloud
  • ต้องกำหนด NTE Ceiling (เพดานค่าใช้จ่ายสูงสุด)

  การจัดการงบประมาณ

  • ยื่นขอ Cloud Budget ผ่านระบบ DGA Portal
  • แต่ละโครงการต้องระบุต้นทุนแยกรายการ
  • ใช้ CSP Pricing Calculator เป็นเอกสารประกอบการจัดซื้อ

  หลักการ FinOps ภาครัฐไทย 4 ด้าน

1. การมองเห็น (Visibility)

แสดง Dashboard ต้นทุนแบบ Real-time ให้ทุกทีมเข้าถึงได้ เพื่อระบุค่าใช้จ่ายที่ไม่จำเป็นได้ทันที

2. การทำงานร่วมกัน

IT รับผิดชอบจัดการทรัพยากร
การเงินรับผิดชอบวางแผนงบ
ทั้งสองต้องสื่อสารกัน

3. การเพิ่มประสิทธิภาพ

ลบทรัพยากรที่ไม่ใช้งาน ใช้ Spot/Reserved Instance ออกแบบระบบให้รองรับ Auto-scaling

4. การวัดผลและวางแผน

กำหนด KPI ติดตามการประหยัด
พยากรณ์การใช้งานในอนาคต
จากข้อมูลในอดีต

6 หลักการ FinOps

แนวทางหลักของ FinOps Foundation

  หลักการที่ 1–3

01

  ทุกทีมต้องทำงานร่วมกัน

การเงิน, IT, ผู้พัฒนา, และผู้บริหาร ต้องทำงานร่วมกันด้วยความเร็วและความละเอียดที่เหมาะสม

02

  คุณค่าทางธุรกิจนำการตัดสินใจ

ตัดสินใจด้วย Unit Economics และ KPI ที่วัดผลได้ มากกว่าการดูแค่ยอดรวมค่าใช้จ่าย

03

  ทุกคนเป็นเจ้าของการใช้ทรัพยากร

ผลักความรับผิดชอบไปถึงทีมที่ใช้งาน — นักพัฒนาต้องคำนึงถึงต้นทุนตั้งแต่ขั้นออกแบบ

  หลักการที่ 4–6

04

  ข้อมูล FinOps ต้องแม่นยำและทันเวลา

ประมวลผลและแบ่งปันข้อมูลต้นทุนทันทีที่มี — การมองเห็นแบบ Real-time ช่วยให้ตัดสินใจได้รวดเร็วขึ้น

05

  FinOps ต้องมีศูนย์กลางขับเคลื่อน

ทีม FinOps กลาง (CCoE) กำหนดมาตรฐาน แนวปฏิบัติ และจัดการ Rate Optimization — ทีมงานรับผิดชอบการใช้งาน

06

  ใช้ประโยชน์จากรูปแบบต้นทุนแบบแปรผัน

Cloud ช่วยให้จ่ายตามที่ใช้จริง — ออกแบบระบบให้ยืดหยุ่น วางแผนแบบ Agile แทนการซื้อล่วงหน้าแบบ Static

  Inform — สร้างความโปร่งใส

ขั้นที่ 1 จาก 3

  Allocate

กำหนด Tags/Labels ให้ทรัพยากร Cloud ทุกชิ้น เพื่อระบุว่าต้นทุนเป็นของทีมหรือโครงการใด

  Report

สร้าง Dashboard แสดงข้อมูลต้นทุนที่นำไปใช้งานได้จริง ทุกทีมเข้าถึงได้

  Forecast

พยากรณ์ค่าใช้จ่ายจากแนวโน้ม
เพื่อวางแผนงบประมาณแม่นยำ

  สิ่งที่ต้องทำก่อน: การระบุต้นทุน (Allocation) ให้ครบถ้วนคือความท้าทายอันดับต้นๆ ของ FinOps — ถ้าไม่รู้ว่าใครใช้เท่าไร ก็เพิ่มประสิทธิภาพไม่ได้

  เครื่องมือ GCP:
Billing Labels, Cost Table Report,
Looker Studio, Budget & Alerts

  KPI ตัวอย่าง:
% Resources มี Labels
Cost per Project, Forecast Accuracy

  Optimize & Operate

Optimize — ขั้นที่ 2
  •   Product Efficiency: Rightsizing, ปิด Resource ที่ไม่ใช้, Refactoring
  •   Pricing Efficiency: CUDs, Preemptible/Spot VM, Custom VM
  •   Guardrails: Budget Alert, Quota, Cost Cap เพื่อป้องกันค่าใช้จ่ายบานปลาย

  ลำดับความสำคัญ: วิเคราะห์ค่าใช้จ่ายที่มีอยู่ → เป้าหมายธุรกิจ → ความคุ้มค่า

Operate — ขั้นที่ 3
  •   Onboard: ออกแบบระบบใหม่ให้คำนึงถึงต้นทุน
  •   Educate: อบรม FinOps ให้ทุกทีมอย่างต่อเนื่อง
  •   Incentivize: ให้รางวัลทีมที่ประหยัดได้
  •   Measure: Unit Economics วัด ROI ต่อผู้บริหาร

  สำคัญ: อย่ารอจนสมบูรณ์แบบ — เริ่มจากเล็กๆ แล้วปรับปรุงไปเรื่อยๆ

บทบาทและความรับผิดชอบ

ใครทำอะไรใน FinOps?

  บทบาทหลักใน FinOps (Personas)

IT / นักพัฒนา

ออกแบบ ติดตั้ง และดูแลระบบ Cloud — รับผิดชอบต้นทุนตั้งแต่ขั้นออกแบบ

การเงิน / งบประมาณ

วางแผนงบ จัดสรรต้นทุน รายงานผล — ต้องเข้าใจรูปแบบค่าใช้จ่าย Cloud

จัดซื้อจัดจ้าง

เจรจาสัญญา จัดการ Reserved/Committed Use ให้ได้ราคาดีที่สุด

IT Operations

บริหารโครงสร้างพื้นฐาน วางแผน Capacity สมดุลระหว่างความเสถียรและต้นทุน

ผู้บริหารโครงการ

เชื่อมโยงต้นทุน Cloud กับ P&L ของโครงการ กำหนดลำดับความสำคัญ

ผู้บริหาร / CCoE

กำหนดทิศทางกลยุทธ์ สร้างวัฒนธรรม FinOps ตั้งแต่ระดับบน

  เริ่มต้น FinOps ในหน่วยงานของท่าน

Crawl

เริ่มจาก Tagging ทรัพยากรและสร้าง Dashboard ต้นทุนพื้นฐาน

Walk

ระบุและลบ Resource ที่ไม่ใช้งาน ตั้ง Budget Alert

Run

Rightsizing, Reserved Instance, Automation FinOps เต็มรูปแบบ

Optimize

Unit Economics, Policy-driven FinOps, CCoE ดำเนินงานเต็มรูป

  Quick Wins ที่ทำได้ทันที

  • ตรวจสอบ Resource ที่ไม่มีการใช้งาน (Idle)
  • กำหนด Labels/Tags ให้ทรัพยากรทุกชิ้น
  • ตั้ง Budget Alert ป้องกันค่าใช้จ่ายเกิน

  ข้อกำหนด DGA สำหรับภาครัฐ

  • ยื่น Cloud Budget ผ่าน DGA Portal
  • กำหนด NTE Ceiling ด้วย CSP Calculator
  • ประเมิน TCO ก่อนตัดสินใจ Private Cloud

  สรุปสาระสำคัญ

FinOps คือการบริหารต้นทุน Cloud อย่างมีความรับผิดชอบ — เป้าหมายคือ คุณค่าสูงสุด ไม่ใช่แค่ลดค่าใช้จ่าย

ภาครัฐไทย บังคับ ใช้ FinOps ตาม DGS 9-2:2568 ภายใต้นโยบาย Go Cloud First ของ ครม.

วงจร Inform → Optimize → Operate ทำซ้ำต่อเนื่อง ไม่ใช่โครงการครั้งเดียว

  3 สิ่งที่ทำได้วันนี้

  •   เริ่ม Tag ทรัพยากร Cloud ทุกชิ้น
  •   ตั้ง Budget Alert ป้องกันค่าใช้จ่ายเกิน
  •   สร้าง ช่องทางสื่อสาร ระหว่าง IT และการเงิน
