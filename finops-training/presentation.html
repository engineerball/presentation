Cloud Cost Management Principles

  FinOps คืออะไร?

"การนำความรับผิดชอบทางการเงินมาใช้กับโมเดลการใช้จ่ายแบบ variable ของคลาวด์ ช่วยให้ทีมต่างๆ ตัดสินใจ trade-off ระหว่าง ความเร็ว, ต้นทุน, และคุณภาพ ได้อย่างแม่นยำ"

— FinOps Foundation

FinOps ≠  การลดค่าใช้จ่ายให้เหลือน้อยที่สุด
FinOps =  การได้ มูลค่าสูงสุด จากทุกบาทที่จ่าย

20% ประหยัดเฉลี่ยที่ย้าย Cloud
40% Cloud cost สูญเปล่าโดยเฉลี่ย
↑↑↑ Cost เพิ่มขึ้นถ้าไม่จัดการ

Source: FinOps Foundation, McKinsey & Atlassian

  6 หลักการ FinOps Framework

1. ทีมต้องร่วมมือกัน

Finance, Engineering, Product ทำงานร่วมกัน

2. มูลค่าธุรกิจ > ต้นทุน

Unit economics สำคัญกว่า aggregate spend

3. ทุกคนรับผิดชอบการใช้งาน

Engineering owns cost ตั้งแต่ design ถึง ops

4. ข้อมูลโปร่งใส ทันเวลา ถูกต้อง

Real-time visibility, fast feedback loops

5. Centralized Enablement

ทีมกลางดูแล rate optimization, best practices

6. ใช้ประโยชน์จาก variable cost

Agile planning แทน static budget

Source: FinOps Foundation — finops.org/framework/principles

  Centralized Enablement + Decentralized Execution

FinOps ทำงานแบบ ทีมกลาง enable, ทีมย่อย execute — ไม่ใช่ทีมกลางทำทุกอย่างเอง

Layer หน้าที่รับผิดชอบ ตัวอย่าง
  Central FinOps Rate optimization, executive comms, best practices, governance negotiate contract, CUDs
  Engineering Usage optimization, architecture decisions, cost-aware design rightsizing, refactor code
  Data Platform Real-time cost visibility, forecasting, benchmarking dashboards, alerts, FOCUS data

  Anti-pattern: ทีมกลางทำ optimization เองทั้งหมด → กลายเป็น bottleneck, ไม่มี ownership

  FinOps Lifecycle — Inform → Optimize → Operate

Inform — มองเห็น

Visibility ของทุก cloud spend (license, usage, IT time), จัดสรร cost, benchmark, report

Optimize — ลด waste

Rightsizing, CUDs/reserved, ปิด resource ที่ไม่ได้ใช้, automation, refactor

Operate — อัตโนมัติ

Continuous improvement, automation, culture change, innovation enablement

  Key Insight: Top-performing teams ไม่หยุดที่ Operate — วนกลับไป Inform & Optimize ต่อไป เพื่อ improve อย่างต่อเนื่อง

  5 Best Practices (Atlassian)

  1. วางแผน FinOps ก่อน ย้าย Cloud

ออกแบบ strategy ตั้งแต่เริ่ม, cloud billing ต่างจาก on-prem — ซื้อเท่าที่จำเป็น, license auto-expire

  2. อย่า trade คุณภาพเพื่อประหยัด

cheapest ≠ best value; trial 3 เดือนประหยัดจริง แต่ paid plan อาจให้ feature/support ที่คุ้มค่าที่สุด

  3. คำนวณ TCO/ROI จริง

ยากกว่าที่คิดในองค์กรใหญ่ siloed teams; forecast + audit เป็นประจำ

  4. ทำให้เป็น ongoing practice

ไม่ใช่ one-time setup — ตั้ง ongoing roles, audits, forecasting

  5. ตั้งคนรับผิดชอบชัดเจน

"If nobody owns it, it goes by the wayside" — freed-up IT capacity เป็นจุดเริ่มต้นที่ดี

Source: Atlassian Blog — atlassian.com/blog/platform/what-is-finops

CHAPTER 2

Cost Allocation และ Budgeting

Trace, Tag, Budget, and Control

  ทำไม Cost Allocation ถึงสำคัญ?

  ปัญหาก่อนทำ

  • Cloud bill มาเป็นก้อนเดียว ไม่รู้ว่าใครใช้
  • optimize ไม่ได้ถ้าไม่เห็นว่า spend ตรงไหน
  • จัดสรร budget ไม่ได้, ไม่มี accountability
  • Shadow IT ซ่อนอยู่ใน shared resources

  เป้าหมายหลังทำ

  • ทุกบาท trace ได้ — รู้ทีม/โปรเจกต์/usage
  • optimize ได้แม่นยำตาม data
  • จัด budget ตาม business outcome
  • สร้าง accountability ทุก layer

  Cost Allocation = การทำให้เห็นภาพจริง ว่าเงิน cloud หมดไปกับอะไร ใครใช้ คุ้มค่าไหม

  Tagging / Labeling Strategy

  Tags/Labels คือหัวใจของ Cost Allocation — ถ้าไม่ tag ก็มองไม่เห็นถ้า tag ไม่ดี ข้อมูลก็ใช้ไม่ได้

  ตัวอย่าง Tag Structure ที่แนะนำ

Tag Key ค่าตัวอย่าง วัตถุประสงค์
cost_center engineering-42 ระบุทีม/แผนกที่รับผิดชอบ
project checkout-v2 ระบุโปรเจกต์
environment dev / staging / prod ระบุสภาพแวดล้อม
application api-gateway ระบุ service/app
owner @team-backend บุคคลหรือทีมรับผิดชอบโดยตรง

  Best Practice: บังคับใช้ tag policy (denied ถ้าไม่มี tag) ตั้งแต่สร้าง resource ใหม่

  Shared Cost — จัดสรรค่าใช้งานทรัพยากรร่วม

เมื่อหลายทีมใช้ resource เดียวกัน (VPC, monitoring, CDN, networking) จะแบ่ง cost ยังไง?

Even Split

แบ่งเท่าๆ กัน
+ ง่าย
- ไม่ fair, ไม่ motivate

Usage-based

แบ่งตาม % ที่ใช้จริง
+ fair, motivate ประหยัด
- ซับซ้อน, ต้องมี data

Fixed %

กำหนด % ตายตัว
+ Predictable
- ไม่สะท้อน actual use

  ตัวอย่าง: CloudFront Cost

ทีม A traffic 60%, ทีม B 30%, ทีม C 10% → แบ่ง bill ตามสัดส่วน 60:30:10

  Budgeting แบบ FinOps

  Traditional Budget

  • Static รายปี
  • อนุมัติตาม resource type
  • เปลี่ยนยาก, review รายปี

  FinOps Budget

  • Dynamic, consumption-based
  • อนุมัติตาม business outcome
  • Review รายเดือน/ไตรมาส

  Unit Economics > Total Cost

ไม่ควรดูแค่ "cloud bill เดือนละเท่าไร" แต่ดู cost per customer, cost per transaction, cost per feature — นี่คือ metric ที่วัดมูลค่าธุรกิจได้จริง

  Forecasting + Actual

  • ใช้ forecast จาก cloud provider + historical trend
  • Compare budget vs actual รายสัปดาห์ ไม่ใช่รายเดือน
  • ปรับ forecast เมื่อ workload เปลี่ยน

  Dashboard & Tooling

  • GCP: Budgets & Alerts, Cost Table, BigQuery billing export
  • AWS: AWS Budgets, Cost Explorer, CE billing anomaly detection
  • Azure: Cost Management + Billing budgets

  Anti-pattern: Finance ดู report เดือนที่แล้ว — Feedback มาช้าเกินแก้! ต้องใช้ real-time หรือ near-real-time data

  FOCUS — FinOps Open Cost & Usage Specification

มาตรฐานจาก FinOps Foundation สำหรับ cost data ที่ vendor-independent — ทำให้ cost data จาก cloud หลายเจ้ามี format เดียวกัน

  ปัญหาก่อน FOCUS

  • AWS, GCP, Azure แต่ละเจ้าใช้ format ต่างกัน
  • รวม cost report ยาก — column name ต่างกัน
  • Tag field คนละชื่อ, resource ID format ต่าง
  • ต้องเขียน custom parser สำหรับแต่ละ provider

  ได้ FOCUS แล้ว

  • Column เดียวกัน ทุก provider (BilledCost, ChargeCategory, Provider...)
  • รวม multi-cloud report ได้ทันที
  • เครื่องมือ FinOps 1 ตัว ใช้ได้ทุก cloud
  • Benchmark cross-cloud ได้จริง

  FOCUS Fields หลัก

BilledCost ChargeCategory Provider ResourceId Region ServiceName ChargeStartTime

CHAPTER 3

Rightsizing Strategies

ปรับขนาด Resource ให้ตรงงาน — ไม่มากไป ไม่น้อยไป

  Rightsizing คืออะไร?

การจับคู่ resource specification ให้ตรงกับ actual workload — "Right" ไม่ใช่ "Smallest" แต่คือ "เหมาะสมที่สุด"

  ปัญหาที่แก้ไข

  Under-provisioned (เล็กเกิน)

  • Performance ตก, latency สูง
  • Downtime, user experience แย่
  • Auto-scale ต้องทำงานหนัก

  Over-provisioned (ใหญ่เกิน)

  • จ่ายสำหรับ CPU/RAM ที่ไม่ได้ใช้
  • Wasted spend 30-40% ของ cloud cost
  • เป็น quick win ที่มี impact สูงสุด

  เป้าหมาย: Sweet Spot (40-70% utilization)

  ทำไมต้อง Rightsize?

40% Cloud cost สูญเปล่า (ไม่ได้ใช้จริง)
$30B+ Wasted cloud spend ทั่วโลก/ปี

  Rightsizing = Quick Win #1 ใน Optimize phase — ได้ ROI สูงสุด ทำง่าย เห็นผลเร็ว

  ประเภททรัพยากรที่ควร Rightsize

Resource Metric ที่ดู วิธี Rightsize
  Compute
VM / Container		 CPU, Memory utilization ลด vCPU/RAM ให้ตรงกับ actual use; ใช้ spot/preemptible สำหรับ non-prod
  Storage
Disk / Object		 Access frequency, IOPS เลือก tier (hot → warm → cold → archive) ตามความถี่เข้าถึง
  Database
RDS / Cloud SQL		 Query load, connections ลด instance class, optimize storage, ใช้ read replica เฉพาะเมื่อจำเป็น
  Network
LB, CDN, Transfer		 Data transfer volume, peak hours optimize data path, ใช้ CDN ลด egress, รวม cross-region traffic

  Metrics ที่ต้องใช้

ไม่ดูแค่ average อย่างเดียว — ต้องดู p95, p99 ด้วย เพื่อไม่พลาด peak time

  Rule of Thumb: ถ้า resource ใช้ < 20% ติดกัน 2 สัปดาห์ → ควรลด size ทันที!

  Key Point: p95/p99 สำคัญกว่า avg — ทรัพยากรที่ avg ใช้ 15% แต่ peak (p99) ใช้ 95% อาจไม่ควร downsize

  Rightsizing Tools

  Cloud-Native

  • GCP: Recommender API — CPU, memory, idle VM, CUD recommendations
  • AWS: Compute Optimizer — EC2, EBS, Lambda, Auto Scaling
  • Azure: Advisor — VM rightsizing, reserved instances

  Third-party

  • Spot.io: Automated right-sizing + spot instance optimization
  • CloudHealth (VMware): Cost + compliance dashboard
  • Cloudability (Apptio): Multi-cloud cost visibility

  Custom

  • Prometheus + Grafana — monitor + dashboard
  • Datadog — APM + infra monitoring
  • เขียน script + alert rules เอง

  ใช้ recommender เป็น ข้อมูลประกอบ — ไม่ใช่ auto-apply ทันที!

  Continuous Rightsizing Process

1

Monitor

เก็บ metrics utilization อย่างต่อเนื่อง

2

Analyze

หา resource ที่ under/over-utilized

3

Recommend

เสนอ size change พร้อม risk assessment

4

Approve

Owner review + approve ก่อน apply

5

Implement

Apply change — ใช้ maintenance window ถ้า prod

6

Verify

ยืนยัน performance OK

  วงจรนี้ต้องรัน ต่อเนื่อง — workload เปลี่ยนตลอดเวลา ไม่ใช่ทำครั้งเดียวจบ

  Anti-patterns ที่ควรเลี่ยง

  Rightsize แล้วไม่ monitor ตาม — เปลี่ยนไป performance แย่ไม่มีใครรู้ จนกว่าจะ outage

  ดูแค่ avg metric — ignore peak time, workload ที่ต้อง burst capacity จะล้มทันที

  Auto-downsize ทุกอย่าง — ไม่มี human review, ไม่มี risk assessment; บาง service ต้อง keep headroom

  ทำครั้งเดียวจบ — workload เปลี่ยนตาม season, feature launch, user growth; ต้องทำเป็น recurring process

  ไม่สื่อสารกับ engineering team — FinOps team สั่ง rightsize โดยไม่คุยกับคนใช้ → trust หาย, collaboration พัง

บทเรียนจากทีม FinOps ทั่วโลก — อย่าทำผิดซ้ำ!
