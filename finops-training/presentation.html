รู้จัก FinOps

FinOps คืออะไร | ใครรับผิดชอบ | เริ่มเมื่อไร

  ทำไมต้องมี FinOps?

  ปัญหาที่พบบ่อยในภาครัฐ: ค่าใช้จ่ายคลาวด์บานปลาย, ไม่มีความโปร่งใส, ทีมต่าง ๆ ไม่รับผิดชอบต่อต้นทุน

Cloud เปลี่ยน Cost Model:
CAPEX ลงทุนครั้งเดียว  →  OPEX จ่ายตามการใช้งาน (Pay-as-you-go)

ความเสี่ยงเมื่อไม่มี FinOps:

  • "Spend Panic" — บิลมาทีหลัง ไม่รู้ว่า spend ตรงไหน
  • Shadow IT — บริการที่ไม่ได้ติดตาม, ไม่ได้ออกแบบ
  • บิลพุ่งโดยไม่มีคนควบคุมหรือรับผิดชอบ

  ข้อมูล: กว่า 70% ขององค์กรที่ migrate ขึ้น cloud โดยไม่มี FinOps พบว่าค่าใช้จ่ายเกิน budget 30-50% ใน 6 เดือนแรก

  FinOps คืออะไร?

"Cloud FinOps is an operational framework and cultural shift that brings technology, finance, and business together to drive financial accountability and accelerate business value realization."

— Google Cloud FinOps Framework

  เปรียบเทียบ FinOps กับ DevOps: DevOps ทลาย Silo ระหว่าง Dev กับ Ops — FinOps ทำแบบเดียวกันแต่เป็น Tech + Finance + Business

  เป้าหมายหลัก: Maximize Cloud Value

ไม่ใช่แค่ "ลดค่าใช้จ่าย" แต่คือ "ลงทุนอย่างฉลาด" เพื่อให้ทุกบาทที่จ่ายสร้างมูลค่าสูงสุด

  ใครควรรับผิดชอบ FinOps?

FinOps Central Team (ทีมกลาง) ทำหน้าที่ "enable" ไม่ใช่ "ควบคุม"

People

จัดตั้ง FinOps Team ภายในกรม/หน่วยงาน — กำหนดบทบาทชัดเจน

Process

กำหนดกระบวนการจัดสรรต้นทุน, ตรวจสอบ, รายงาน

Technology

ใช้เครื่องมือ GCP: Billing, BigQuery, Looker Studio

  Anti-pattern ในภาครัฐ: ทีม FinOps กลายเป็นทีมกลางที่ทำทุกเรื่อง — ไม่มี distributed accountability

  เมื่อไรควรเริ่มใช้ FinOps?

  เริ่มตั้งแต่ต้น — ไม่ใช่รอให้บิลพุ่ง

ควรเริ่มคู่กับ Cloud Transformation Program:

    6 หลักการวัฒนธรรม FinOps

จาก Google Cloud FinOps Getting Started Guide

1. Variable Nature

Cloud cost ผันแปรตามการใช้งาน

2. Business-Value Decision

ตัดสินใจจากคุณค่า ไม่ใช่แค่ราคาถูก

3. Accountability

ทุกคนที่ใช้คลาวด์รับผิดชอบต้นทุน

4. Collaboration

6. Democratization of Data

ข้อมูลต้นทุนโปร่งใส เข้าถึงได้

    หลักการที่ 1: Variable Nature of Cloud

ต้นทุน Cloud ผันแปรตามการใช้งาน (Pay-as-you-go) — ต่างจาก on-premise ที่จ่ายคงที่ล่วงหน้า

  Mindset เก่า: ระบบต้องเปิด 24/7 = เสถียร

  Mindset ใหม่: Server ที่ปิดอยู่ = ประหยัดต้นทุน

  ตัวอย่างสำหรับภาครัฐ

  • ระบบลงทะเบียนออนไลน์ — ใช้เฉพาะช่วงเปิดรับ
  • ระบบยื่นภาษี — พีคช่วงฤดูภาษี
  • ระบบเลือกตั้ง/สำรวจ — ใช้งานเป็นช่วง ๆ
  • → ใช้ auto-shutdown, preemption หรือ scaling เพื่อลดต้นทุนนอกช่วงใช้งาน

    หลักการที่ 2: Business-Value Based Decision Making

ตัดสินใจด้านต้นทุนจากมุมมอง "คุณค่าทางธุรกิจ" — ไม่ใช่แค่ "ถูกที่สุด"

  IaaS vs Managed Services

Managed Services อาจแพงกว่า IaaS แต่ ประหยัดค่าบริหารจัดการ, ลด downtime, ทีมเล็กก็ดูแลได้

  สำหรับภาครัฐ

  • ROI ของบริการดิจิทัลต่อประชาชน — ประหยัดค่าเดินทาง, ลดเวลา, เพิ่มความโปร่งใส
  • Cloud ช่วยให้รัฐบาลดิจิทัล (e-Government) ใช้งานข้อมูลข้ามหน่วยงานได้
  • ต้องวัด business value ไม่ใช่แค่ infrastructure cost

    หลักการที่ 3-6

  Accountability

ทุกคนที่ใช้ Cloud มีส่วนรับผิดชอบต่อต้นทุน — ไม่ใช่แค่ทีม IT

  Collaboration

ทีม IT + การเงิน + ผู้ใช้งานต้องทำงานร่วมกัน — ไม่ใช่ต่างคนต่างทำ

  Blamelessness

เมื่อเกิดค่าใช้จ่ายเกิน → หาสาเหตุ ไม่ใช่หาคนผิด — สร้างวัฒนธรรมปลอดภัยในการเรียนรู้

  Democratization of Data

ทุกคนควรเห็นข้อมูลต้นทุน — ตัดสินใจได้ดีขึ้น โปร่งใส ตรวจสอบได้

FinOps Lifecycle บน GCP

Inform → Optimize → Operate (FinOps Foundation / Google)

    FinOps Lifecycle — 3 Phase

Inform

จัดสรร, รายงาน, พยากรณ์

ทำให้มองเห็น

Optimize

Rightsizing, Pricing, Guardrails

เพิ่มประสิทธิภาพ

Operate

Onboard, Educate, Measure

ฝังวัฒนธรรม

  Lifecycle นี้เป็น วงจรต่อเนื่อง — ทีมที่ดีจะวนกลับมา Inform และ Optimize ซ้ำ ๆ เพื่อ improve ตลอดเวลา

    Phase 1: Inform (ทำให้มองเห็น)

Allocate

จัดสรรต้นทุนด้วย Labels/Tags บน GCP

  • กำหนด label: agency, project, env, owner
  • จัดสรร shared cost

Report

Cloud Billing Export → BigQuery → Looker Studio

  • Dashboard รายเดือน
  • Compare trend
  • KPI per project/service

Forecast

พยากรณ์ค่าใช้จ่ายล่วงหน้า

  • ใช้ historical data + trend
  • เตือนล่วงหน้าก่อนงบประมาณเกิน

    Phase 2: Optimize (เพิ่มประสิทธิภาพ)

  Product Efficiency

  • Rightsizing: ปรับ size VM, DB, storage ให้ตรง workload
  • Refactoring: ย้าย code ให้ใช้ serverless/container

  Pricing Efficiency

  • Committed Use Discounts (CUD): ประหยัดถึง 57%
  • Spot VMs: ประหยัดถึง 91%
  • Sustained Use Discounts: ลดอัตโนมัติ

  Guardrails

  • Budget Alerts: แจ้งเตือนเมื่อใกล้เกินงบ
  • Quota Management: จำกัดการ provisioning resource เกินกำหนด

    Phase 3: Operate (ฝังวัฒนธรรม)

  Onboard Workloads

กำหนดมาตรฐาน labeling/ตั้งแต่เริ่มสร้าง resource — "ถ้าไม่มี label = ไม่อนุมัติ deploy"

  Educate & Enable

อบรมทีมงานทุกระดับ — ตั้งแต่ CIO, ผู้บริหาร, technical team, finance — ให้มี cost awareness

  Incentivize & Measure Value

สร้างแรงจูงใจให้ทีมประหยัดได้ — วัดผลตาม unit economics, cost per transaction, ROI ของบริการต่อประชาชน

จัดสรร, ติดป้าย, ตั้งงบประมาณ, รายงานต้นทุน

    แนวคิด Cost Allocation

ทำไมต้องจัดสรรต้นทุน? โปร่งใส, สร้าง Accountability, วางแผนงบประมาณได้แม่นยำ

  ความท้าทายของภาครัฐ: งบประมาณรายปี (fixed) vs. ต้นทุนคลาวด์แบบ Variable — ไม่ match กัน!

แนวทางการจัดสรรในภาครัฐ:

  • ตั้ง budget รายไตรมาส ไม่ใช่รายปี — ปรับทุกไตรมาส
  • ใช้ reserve buffer 10-20% ของ budget สำหรับ fluctuation
  • แยก project budget ออกจากกัน — ไม่ปนกัน

    Shared Cost Allocation

Shared Resources: VPC, Monitoring, Logging — หลายกรมใช้ร่วมกัน จะจัดสรรยังไง?

Even Split

แบ่งเท่า ๆ กัน
+ ง่าย - ไม่ fair

Proportional

แบ่งตาม % การใช้
+ fair - ซับซ้อน

ระบุตรงว่าใครใช้
+ แม่นยำ

  กรณีภาครัฐ

ระบบที่หลายกรมใช้ร่วมกัน — e.g. Government Network (GovNet), Central Logging, Shared Identity — ควรใช้ Proportional split ตาม % traffic/users ของแต่ละกรม

Rightsizing Strategies

ปรับขนาด Resource ให้ตรงกับ Workload จริง

    Rightsizing คืออะไร?

การปรับขนาด Resource ให้เหมาะสมกับ Workload จริง

  Over-provisioning
จ่ายเกินความจำเป็น — ค่าใช้จ่ายสูญเปล่า

  Under-provisioning
ประสิทธิภาพต่ำ — User experience แย่, downtime

   เป้าหมาย: "Right" = "ไม่มากไป ไม่น้อยไป" — เหมาะสมที่สุดกับงาน

    ปัจจัยในการพิจารณา Rightsizing

ประเภท Workload

  • Stateful / Stateless
  • Batch / Realtime
  • Latency-sensitive?

ขนาด Workload

  • Traffic volume
  • Data size
  • Concurrent users

ความต้องการ Performance

  • CPU / Memory
  • IOPS / Throughput
  • Availability (SLA)

    Rightsizing ตาม 8 กลุ่มบริการ GCP

กลุ่ม 策略
Compute
  • ใช้ Recommender → Resize VM, เปลี่ยน machine family (e2/n2/c2)
  • App Engine: ปรับ Scaling (Automatic/Basic/Manual)
  • CUD (1/3 ปี) สำหรับ Baseline workload
Storage
  • Storage Class: Standard → Nearline → Coldline → Archive
  • Disk Type: pd-standard / pd-ssd / pd-balanced ตาม IOPS
Networking
  • ใช้ Cloud CDN ลด egress, จัด resources ใน Region เดียวกัน
  • Cloud Interconnect สำหรับ On-prem ปริมาณสูง

    Rightsizing (ต่อ)

Database
  • Cloud SQL: ปรับ tier, เปิด Auto Storage Increase, Read Replicas
  • Cloud Spanner: ปรับ Processing Units
  • Bigtable: ปรับ Node ตาม Throughput
Analytics
  • BigQuery: On-demand vs. Slot Reservations, Partitioning + Clustering
  • Dataflow: ปรับ Worker, ใช้ Autoscaling
Dev/Mgmt
  • กำหนด Retention policy ของ Metrics/Logs
  • ลด ingestion ด้วย Log Exclusion filters
Security สคค SCC Tier (Standard/Premium/Enterprise) ให้เหมาะสม
Others ทบทวน Marketplace subscriptions ที่ไม่ได้ใช้งาน

    Pricing Efficiency — ลดต้นทุนด้วยราคา

  Committed Use Discounts (CUD)

ประหยัดถึง 57% สำหรับ Compute — ยอมผูก 1 ปี หรือ 3 ปี สำหรับ workload ที่รันตลอด

  Spot VMs (Preemptible)

ประหยัดถึง 91% — เหมาะสำหรับ Batch, Fault-tolerant, ML training

  Sustained Use Discounts

ลดอัตโนมัติเมื่อใช้งานต่อเนื่องภายในเดือน — ไม่ต้องทำอะไร

  BigQuery Flat-rate

Slot Reservations — เหมาะสำหรับ workload ปริมาณสูงและสม่ำเสมอ

    Guardrails — ป้องกันค่าใช้จ่ายที่ไม่คาดคิด

  Organization Policy

จำกัด Region — ป้องกัน deploy Region แพงโดยไม่รู้ตัว, จำกัด Machine Type สูงสุด

  Budget Alerts + Pub/Sub + Cloud Functions

แจ้งเตือน → Automate response → shutdown resource อัตโนมัติเมื่อเกิน threshold

  Quota Management

กำหนดขอบเขตการใช้ Resource — จำกัด CPU cores, disk size, number of instances ต่อ Project

  แนวปฏิบัติ: เริ่มด้วย alert ก่อน → พอ trust workflow แล้วค่อยเปิด auto-action
